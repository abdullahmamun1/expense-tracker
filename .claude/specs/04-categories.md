# Spec: Categories

## Overview

The second data-bearing feature of the expense tracker: users define named "categories" (e.g.
Groceries, Rent, Salary) used to classify money movements, each tagged as either an expense or
income category and given a color swatch for quick visual scanning in later ledger views. Like
wallets, categories are private per-user data with no relationship to any other model yet — the
upcoming transaction-logging step will reference both a wallet and a category, so both must exist
first. This step follows the exact CRUD/route-guard/module shape established by wallets in step 03,
just with a second, independent resource.

## Depends on

- Step 01 (Login and Signup) — categories are private per-user data; every category route requires
  an authenticated session (`req.session.userId`), and the frontend page requires a logged-in user.
- Step 03 (Accounts Wallet) — not a functional dependency (categories do not reference wallets),
  but this step reuses the wallet feature's module structure, route-guard (`proxy.ts`), nav-link
  pattern, and ledger card styling as its direct template.

## Routes

Backend (Express, mounted under `/api/categories`, all logged-in only):

- `POST /api/categories` — create a category (name, type, color) — logged-in
- `GET /api/categories` — list the current user's categories — logged-in
- `GET /api/categories/:id` — get a single category owned by the current user — logged-in
- `PATCH /api/categories/:id` — update a category's name/type/color owned by the current user — logged-in
- `DELETE /api/categories/:id` — delete a category owned by the current user — logged-in

Frontend (Next.js App Router):

- `GET /categories` — category list + create/edit/delete UI — logged-in (redirects to `/login` if
  no session cookie is present)

## New dependencies

No new dependencies — reuses the shadcn `select` component already added in step 03 for the
category "type" field, and a plain-CSS color swatch (a small set of preset hex values rendered as
clickable dots) for the "color" field rather than pulling in a color-picker package.

## Rules for implementation

- If a feature needs API implementation, you can add it.
- All templates make ShadcnUi.
- Every category route is protected by the existing `requireAuth` middleware
  (`backend/src/modules/auth/auth.middleware.ts`).
- Every category query must be scoped to `req.session.userId` — never trust a client-supplied
  user ID. Fetching, updating, or deleting a category that doesn't belong to the current session's
  user must return `404` (not `403`, to avoid confirming the category's existence to other users).
- Backend follows the existing feature-module convention: all category files live together under
  `backend/src/modules/categories/`, matching `backend/src/modules/wallets/`.
- Frontend follows the existing "ledger" design system (Fraunces serif + IBM Plex Mono, cream
  paper palette, dashed-divider itemized lists) — no new colors, fonts, or one-off styles beyond
  the category color swatches themselves.
- Prisma is the only DB access layer; the `Category` model is added via `prisma migrate dev`, not
  a hand-written SQL migration.
- Category `color` is stored as a hex string (e.g. `#4c6b52`) validated against a fixed allowlist
  of preset swatches on both the zod schema and the backend validator, so arbitrary CSS/HTML
  cannot be injected through the field.
- A category `name` must be unique per user (case-insensitive) — enforce with a composite unique
  constraint (`userId`, lower-cased `name`) or an application-level check in `category.service.ts`.

## Definition of done

- [ ] `npm run dev` in both `frontend/` and `backend/` starts with no errors
- [ ] `POST /api/categories` without a session cookie returns `401`
- [ ] `POST /api/categories` with a valid session creates a category scoped to that user and
      returns `201`
- [ ] `GET /api/categories` returns only the logged-in user's categories, never another user's
- [ ] `PATCH /api/categories/:id` updates a category owned by the current user; attempting to
      update a category owned by a different user returns `404`
- [ ] `DELETE /api/categories/:id` removes a category owned by the current user; attempting to
      delete another user's category returns `404`
- [ ] Creating a category with a name that already exists for that user (case-insensitive) is
      rejected with a validation error
- [ ] Visiting `/categories` while logged out redirects to `/login`
- [ ] Visiting `/categories` while logged in renders the category list, with a clear empty state
      when the user has no categories yet
- [ ] Creating a category through the form adds it to the list without a full page reload, showing
      its color swatch and type
- [ ] Editing and deleting a category from the UI works and the change persists after reloading
      the page
- [ ] The header's "Categories" link is visible only when authenticated and navigates to
      `/categories`
