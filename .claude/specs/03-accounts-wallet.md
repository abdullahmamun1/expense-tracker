# Spec: Accounts Wallet

## Overview

The first data-bearing feature of the expense tracker: users create one or more "wallets" —
named money containers such as a checking account, a cash wallet, or a credit card — each with a
starting balance. Every expense/income logged in later steps will need to be attributed to a
wallet, so this must exist before transaction-logging features can be built. This is also the
first logged-in-only page in the app (login/signup only ever redirected back to the marketing
homepage), so it introduces the first real route guard for authenticated pages.

## Depends on

Step 01 (Login and Signup) — wallets are private per-user data; every wallet route requires an
authenticated session (`req.session.userId`), and the frontend page requires a logged-in user.

## Routes

Backend (Express, mounted under `/api/wallets`, all logged-in only):

- `POST /api/wallets` — create a wallet (name, type, starting balance) — logged-in
- `GET /api/wallets` — list the current user's wallets — logged-in
- `GET /api/wallets/:id` — get a single wallet owned by the current user — logged-in
- `PATCH /api/wallets/:id` — update a wallet's name/type owned by the current user — logged-in
- `DELETE /api/wallets/:id` — delete a wallet owned by the current user — logged-in

Frontend (Next.js App Router):

- `GET /wallets` — wallet list + create/edit/delete UI — logged-in (redirects to `/login` if no
  session cookie is present)

## New dependencies

- shadcn `select` component (`npx shadcn add select`) for the wallet "type" field — not a
  standalone npm package, but generates a new primitive under `frontend/components/ui/` the way
  `button`/`input`/`card` were generated in step 01.

No new backend dependencies — reuses `express`, `zod`, and `@prisma/client` already installed.

## Rules for implementation

- If a feature needs API implementation, you can add it.
- All templates make ShadcnUi.
- Every wallet route is protected by the existing `requireAuth` middleware
  (`backend/src/modules/auth/auth.middleware.ts`).
- Every wallet query must be scoped to `req.session.userId` — never trust a client-supplied user
  ID. Fetching, updating, or deleting a wallet that doesn't belong to the current session's user
  must return `404` (not `403`, to avoid confirming the wallet's existence to other users).
- Backend follows the existing feature-module convention: all wallet files live together under
  `backend/src/modules/wallets/`, matching `backend/src/modules/auth/`.
- Frontend follows the existing "ledger" design system (Fraunces serif + IBM Plex Mono, cream
  paper palette, dashed-divider itemized lists) — no new colors, fonts, or one-off styles.
- Prisma is the only DB access layer; the `Wallet` model is added via `prisma migrate dev`, not
  a hand-written SQL migration.
- Starting balance is stored as a fixed-point/decimal type in Postgres (not a floating-point
  number) to avoid rounding errors in money math.

## Definition of done

- [ ] `npm run dev` in both `frontend/` and `backend/` starts with no errors
- [ ] `POST /api/wallets` without a session cookie returns `401`
- [ ] `POST /api/wallets` with a valid session creates a wallet scoped to that user and returns
      `201`
- [ ] `GET /api/wallets` returns only the logged-in user's wallets, never another user's
- [ ] `PATCH /api/wallets/:id` updates a wallet owned by the current user; attempting to update a
      wallet owned by a different user returns `404`
- [ ] `DELETE /api/wallets/:id` removes a wallet owned by the current user; attempting to delete
      another user's wallet returns `404`
- [ ] Visiting `/wallets` while logged out redirects to `/login`
- [ ] Visiting `/wallets` while logged in renders the wallet list, with a clear empty state when
      the user has no wallets yet
- [ ] Creating a wallet through the form adds it to the list without a full page reload
- [ ] Editing and deleting a wallet from the UI works and the change persists after reloading the
      page
