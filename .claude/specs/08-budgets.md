# Spec: Budgets

## Overview

The first planning feature on top of the existing ledger: users set a monthly spending limit on
any of their expense categories, and the app tracks how much of that limit has been used by the
current calendar month's transactions. Where the dashboard (step 07) is a passive, read-only
summary, budgets are a per-category target the user opts into and can create/edit/delete — turning
"how much did I spend on Groceries" into "how much of my Groceries budget is left." A budget is an
ongoing monthly limit (not scoped to a single month instance); actual spend is always evaluated
against the current calendar month, using the same month-boundary rule already established by the
dashboard.

## Depends on

- Step 01 (Login and Signup) — budgets are private per-user data; every budget route requires an
  authenticated session (`req.session.userId`), and the frontend page requires a logged-in user.
- Step 04 (Categories) — every budget references an existing `EXPENSE` category owned by the
  current user; the category picker in the budget form is populated from `GET /api/categories`
  filtered to `type: EXPENSE`. Income categories cannot have a budget.
- Step 05 (Transactions) — "actual spend" for a budget is computed by summing the current user's
  transactions in the budgeted category for the current calendar month.
- Not a functional dependency, but reuses the "current calendar month" boundary logic
  (`occurredAt >= first day of month`, `< first day of next month`) established in Step 07
  (Dashboard) so the two features stay consistent.

## Routes

Backend (Express, mounted under `/api/budgets`, all logged-in only):

- `POST /api/budgets` — create a budget (categoryId, monthlyLimit) for one of the user's own
  `EXPENSE` categories — logged-in
- `GET /api/budgets` — list the current user's budgets, each including the computed current-month
  `actualSpent`, `remaining`, and `percentUsed` — logged-in
- `GET /api/budgets/:id` — get a single budget owned by the current user, with the same computed
  fields — logged-in
- `PATCH /api/budgets/:id` — update a budget's `monthlyLimit` owned by the current user — logged-in
- `DELETE /api/budgets/:id` — delete a budget owned by the current user — logged-in

Frontend (Next.js App Router):

- `GET /budgets` — budget list + create/edit/delete UI — logged-in (redirects to `/login` if no
  session cookie is present)

## Rules for implementation

- If a feature needs API implementation, you can add it.
- All templates make ShadcnUi.
- Every budget route is protected by the existing `requireAuth` middleware
  (`backend/src/modules/auth/auth.middleware.ts`).
- Every budget query must be scoped to `req.session.userId` — never trust a client-supplied user
  ID. Fetching, updating, or deleting a budget that doesn't belong to the current session's user
  must return `404` (not `403`, to avoid confirming the budget's existence to other users).
- On create, the referenced `categoryId` must be independently verified as owned by
  `req.session.userId` and must have `type: EXPENSE`; a budget on an `INCOME` category or a
  category owned by another user is rejected (`404` if not owned, `422` if it's an `INCOME`
  category).
- A user may only have one budget per category — enforce with a unique constraint (`userId`,
  `categoryId`) on the `Budget` model; creating a second budget for a category that already has one
  is rejected with a validation error, directing the user to edit the existing budget instead.
- `actualSpent` is never stored on the `Budget` row — it is always computed at read time by summing
  the user's transactions in that category for the current calendar month (same "this month"
  window as the dashboard: `occurredAt >= first day of current month at 00:00` and `< first day of
  next month`), so it always reflects live ledger data.
- `Budget.categoryId` uses `onDelete: Cascade` — deleting a category also deletes its budget (this
  is independent of the existing rule that blocks deleting a category with transactions; a budget
  alone never blocks category deletion).
- Backend follows the existing feature-module convention: all budget files live together under
  `backend/src/modules/budgets/`, matching `backend/src/modules/categories/`.
- Frontend follows the existing "ledger" design system (Fraunces serif + IBM Plex Mono, cream paper
  palette, dashed-divider itemized lists) — no new colors beyond the category's own stored `color`
  for its swatch and the existing `destructive` token for the over-budget state.
- `monthlyLimit` is stored as a fixed-point/decimal type in Postgres (`Decimal(12,2)`, matching
  `Category`/`Transaction` money fields), always a positive value.
- Prisma is the only DB access layer; the `Budget` model is added via `prisma migrate dev`, not a
  hand-written SQL migration.

## Definition of done

- [ ] `npm run dev` in both `frontend/` and `backend/` starts with no errors
- [ ] `POST /api/budgets` without a session cookie returns `401`
- [ ] `POST /api/budgets` with a valid session and an owned `EXPENSE` category creates a budget and
      returns `201`
- [ ] `POST /api/budgets` targeting an `INCOME` category is rejected with a validation error
- [ ] `POST /api/budgets` targeting a category owned by a different user returns `404`
- [ ] `POST /api/budgets` for a category that already has a budget is rejected with a validation
      error
- [ ] `GET /api/budgets` returns only the logged-in user's budgets, each with a computed
      `actualSpent`, `remaining`, and `percentUsed` for the current calendar month
- [ ] `PATCH /api/budgets/:id` updates a budget's `monthlyLimit` owned by the current user;
      attempting to update another user's budget returns `404`
- [ ] `DELETE /api/budgets/:id` removes a budget owned by the current user; attempting to delete
      another user's budget returns `404`
- [ ] Deleting a category that has a budget also removes the budget, with no error
- [ ] Logging a new transaction in a budgeted category updates that budget's `actualSpent` and
      progress bar without any manual recalculation step
- [ ] Transactions dated in a previous calendar month do not count toward the current month's
      `actualSpent` for any budget
- [ ] Visiting `/budgets` while logged out redirects to `/login`
- [ ] Visiting `/budgets` while logged in renders the budget list, with a clear empty state when
      the user has no budgets yet, and guidance to create an expense category first if none exist
- [ ] Creating a budget through the form adds it to the list without a full page reload, showing
      its category, limit, and progress bar
- [ ] A budget whose spend has met or exceeded its limit visually indicates over-budget status
      (`text-destructive`)
- [ ] Editing and deleting a budget from the UI works and the change persists after reloading the
      page
- [ ] The header's "Budgets" link is visible only when authenticated and navigates to `/budgets`
- [ ] Budget data for one user never includes another user's categories or transactions
